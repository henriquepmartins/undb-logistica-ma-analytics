\begin{abstract}
Este documento descreve a solução desenvolvida para responder consultas do tipo \emph{"quantos pacotes atrasaram mais de 30 minutos em uma janela de tempo"} com tempo de resposta inferior a 3 segundos, mesmo com até dois milhões de eventos. Propõe-se um índice baseado em ordenação temporal e prefix sums por hub e status, reduzindo o custo de consulta de $\Theta(n)$ para $\Theta(\log n)$. São apresentados a modelagem do problema, análise de complexidade temporal e espacial, metodologia experimental, resultados de benchmarks e conclusão sobre a conformidade com o SLA do desafio.
\end{abstract}

\section{Introdução}
O desafio consiste em acelerar consultas sobre logs logísticos que, na abordagem ingênua, percorrem toda a base. Em volumetrias acima de $10^6$ registros, o tempo de resposta torna-se incompatível com uso operacional. O objetivo é garantir latência $< 3$ segundos para janelas arbitrárias, mantendo corretude dos resultados quando comparados ao baseline linear.

\section{Requisitos do Desafio}
\begin{itemize}
  \item Responder consultas de atraso $> 30$ minutos em janelas de tempo arbitrárias com SLA de 3s até 2M de eventos.
  \item Permitir filtros por hub e por status (e combinações).
  \item Manter equivalência de resultados com a contagem linear de referência.
  \item Disponibilizar benchmark comparativo e dashboard exploratório.
\end{itemize}

\section{Dataset e Normalização}
A base oficial localiza-se em \texttt{data/logistica\_ma\_logs.csv}, com colunas obrigatórias \{\texttt{id}, \texttt{timestamp}, \texttt{status}, \texttt{hub}, \texttt{atraso\_min}\}. A normalização converte timestamps para \texttt{timestamp\_epoch} (int64) e cria rótulos legíveis. Ingestões CSV/JSON compartilham o mesmo contrato.

\section{Solução Proposta}
\subsection{Pré-processamento}
\begin{enumerate}
  \item Ordenação estável por \texttt{timestamp\_epoch} ($\Theta(n \log n)$) preserva ordem relativa para empates e maximiza locality de memória.
  \item Construção de vetores por escopo: global, por \texttt{hub}, por \texttt{status} e por par (\texttt{hub}, \texttt{status}). Cada escopo armazena: (a) \texttt{timestamps} (int64), (b) \texttt{delays} (int32), (c) \texttt{delayed\_30\_prefix} (int64), (d) \texttt{delay\_sum\_prefix} (int64), (e) \texttt{max\_delay\_prefix} (int32). Custo $\Theta(n)$ por escopo, construído por agrupamento e `cumsum` vetorial.
  \item Estrutura resultante: dicionários chaveados por \texttt{hub}, por \texttt{status} e pelo par (\texttt{hub}, \texttt{status}) apontando para fatias ordenadas e prefixos compartilhados.
\end{enumerate}

\subsection{Consulta (como é respondida)}
\begin{enumerate}
  \item Seleção do índice: global se não houver filtro; índice do hub; índice do status; ou índice do par (hub, status) se ambos forem informados.
  \item Limites temporais: busca binária via \texttt{np.searchsorted} em \texttt{timestamps} ($\Theta(\log n)$) retorna \texttt{left}, \texttt{right}. Se \texttt{left} $\geq$ \texttt{right}, retorna 0 eventos.
  \item Contagem de atrasos: se \texttt{min\_delay} = 30, usa diferença de prefixo \texttt{delayed\_30\_prefix[right] - delayed\_30\_prefix[left]} ($\Theta(1)$). Caso contrário, avalia somente o \texttt{delays[left:right]} com comparação vetorial ($\Theta(k)$, onde $k = \texttt{right-left}$).
  \item Sumário: total de eventos = $k$; atraso médio obtido por diferença de \texttt{delay\_sum\_prefix}; atraso máximo por `max` na janela local; todos os passos operam apenas na fatia temporal selecionada.
\end{enumerate}

\section{Análise de Complexidade}
\subsection{Tempo}
\begin{itemize}
  \item Pré-processamento: $\Theta(n \log n)$ (ordenar) + $\Theta(n)$ (prefixos) por escopo. Em prática, construção para 2M levou $\approx$ 1.32s no hardware de testes.
  \item Consulta com \texttt{min\_delay}=30: $\Theta(\log n)$ (busca) + $\Theta(1)$ (prefixo). Latência medida: $\approx 1\,\mu$s no caso de 2M (arredondamento para 6 casas no benchmark).
  \item Consulta com \texttt{min\_delay}$\neq$30: $\Theta(\log n + k)$, onde $k$ é o tamanho da janela temporal filtrada. Para janelas estreitas, $k \ll n$ e o custo permanece sub-milisegundo.
\end{itemize}
\subsection{Espaço}
\begin{itemize}
  \item Por linha: 1 int64 (timestamp) + 1 int32 (delay) + 2 int64 de prefixo + 1 int32 de max acumulado $\approx$ 8+4+8+8+4 = 32 bytes por linha e por escopo. Como cada linha pertence a um único hub e a um único status, o fator de multiplicação dos índices se mantém próximo de 3 estruturas independentes, não uma duplicação completa de $n$.
  \item Para 2M linhas, o índice global consome \~64 MB; índices por hub/status dividem os dados por grupo, mantendo a soma total na ordem de algumas centenas de MB, compatível com a máquina de teste.
\end{itemize}

\section{Validação e Testes}
\begin{itemize}
  \item \textbf{Corretude de contagem}: comparação índice vs. baseline linear no dataset oficial e em amostras sintéticas.
  \item \textbf{Filtros}: testes unitários para filtros por hub e por status, além de \texttt{min\_delay} variável.
  \item \textbf{Dataset}: ingestão CSV/JSON com contrato comum; dataset oficial carregado sem upload manual.
  \item \textbf{Benchmark CLI}: verificação automática de SLA para $n=2{,}000{,}000$ e checagem de mesma resposta que o baseline.
\end{itemize}

\section{Metodologia Experimental}
\begin{itemize}
  \item \textbf{Hardware}: macOS (CPU Apple Silicon, 16 GB RAM). Execuções single-threaded (numpy/pandas default) sem afinidade específica.
  \item \textbf{Semente}: gerador \texttt{np.random.default\_rng(42)} para reprodutibilidade dos perfis sintéticos.
  \item \textbf{Perfis}: categorias e distribuição de \texttt{hub}, \texttt{status} e \texttt{atraso\_min} extraídas da base oficial (função \texttt{load\_benchmark\_profile}).
  \item \textbf{Janelas}: para cada tamanho, escolhe-se uma janela central: \texttt{start} = índice $n/4$, \texttt{end} = $n/4 + n/2$, garantindo volume significativo e sem viés para extremos.
  \item \textbf{Cenários}: tamanhos \{1k, 10k, 100k, 1M, 2M\}; consulta com \texttt{min\_delay}=30 e comparação com baseline linear. Em testes de regressão, há casos adicionais com \texttt{min\_delay}=45 e filtros hub+status no dataset real.
  \item \textbf{SLA}: o CLI de benchmark lança exceção se o caso de 2M exceder 3s ou divergir do baseline (função \texttt{\_assert\_sla}).
\end{itemize}

Observações adicionais:
\begin{itemize}
  \item \textbf{Speedup}: no maior cenário, o índice foi $\sim$ 62{,}7k vezes mais rápido que o baseline linear na mesma janela.
  \item \textbf{Corretude}: todas as linhas da tabela indicam respostas idênticas ao baseline; o CLI falha se houver divergência.
  \item \textbf{Filtros e limiares}: em testes funcionais, consultas com \texttt{hub+status} e \texttt{min\_delay}=45 no dataset real retornaram valores iguais ao cálculo manual e permaneceram sub-milisegundo.
  \item \textbf{Build}: o custo de construção (1.32s para 2M) ocorre uma vez por carga; consultas subsequentes amortizam o custo.
\end{itemize}

\section{Conclusões}
A estrutura indexada reduz o custo de consulta de $\Theta(n)$ para $\Theta(\log n)$ com constantes pequenas: no benchmark sintético atual, as consultas ficaram na ordem de microssegundos, mesmo em 2M registros, e o SLA de 3 segundos foi atendido com ampla folga. O build em $\approx$ 1.3 s é pago uma vez por carga. Testes automatizados cobrem baseline linear, filtros por hub/status, limiares variados e validam que o índice replica exatamente o resultado de referência.
